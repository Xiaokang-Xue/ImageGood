# 安全策略

## 支持范围

当前安全修复优先应用于 `main` 分支。正式发布策略建立前，历史提交和未列入 GitHub Releases 的版本不单独提供安全维护承诺。

## 私下报告漏洞

请不要在公开 Issue、Pull Request、讨论区或日志截图中披露未修复漏洞。

当前专用安全邮箱尚待维护者配置。维护者应在公开仓库前将下列占位符替换为可用的私密渠道：

```text
security-contact@your-domain.example
```

在专用渠道可用前，请通过仓库所有者 GitHub 主页中公布的非公开联系方式联系，并只提供必要信息。报告建议包含：

- 受影响版本、路由或组件。
- 可复现步骤和影响范围。
- 已脱敏的请求、响应或日志。
- 建议修复方式（可选）。

不要发送真实用户图片、数据库导出、支付私钥或仍有效的生产密钥。

## 密钥泄露

如果 API Key、Webhook、数据库密码、COS 凭据、SMTP / 短信凭据或支付私钥可能泄露，应立即：

1. 在对应服务控制台吊销或轮换凭据。
2. 更新服务器 secret 与 `.env.local`，重启受影响进程。
3. 检查访问日志、支付回调和异常调用。
4. 从 Git 历史和构建产物中清理泄露内容；仅删除最新提交不足以消除泄露。
5. 记录影响范围和后续修复，不在公开渠道再次粘贴原密钥。

## 安全边界

- 浏览器不得接触模型、数据库、COS、短信、邮件和支付密钥。
- 支付到账以服务端异步通知验签、订单号和金额校验为准。
- 用户任务、订单与管理员接口必须在服务端鉴权。
- 生产环境应使用 HTTPS、安全 Cookie、高强度 `AUTH_SECRET` 和最小权限凭据。
